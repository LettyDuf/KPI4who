Seuils / paliers dans le domaine — 3 options

Question ouverte au terme de l'étape (a) du chantier 14. L'inventaire a recensé un écart §5.7 (c) : aucune fiche ne porte aujourd'hui de seuils de référence. La crainte exprimée : qu'une structuration trop zélée impose des cibles universelles là où chaque contexte a les siennes.

« La confiance ne se mesure pas pour la piloter — elle se mesure pour détecter qu'on la perd. »
— Jurgen Appelo, repris dans la fiche s8 de l'outil.

Trois options structurées ci-dessous, appliquées à deux fiches-test : o1 (Fréquence de déploiement DORA — repères industriels reconnus existent) et s8 (Indice de confiance organisationnelle — aucune cible légitime).

A Zéro seuil dans le domaine

Le cœur décrit l'intention et les anti-patterns. Il ne porte aucune cible, aucune bande, aucune graduation. Si l'équipe veut des paliers, elle les pose à côté — dans son Notion, son tableau de bord, son propre rituel.

Avantage
Pureté du domaine. Zéro imposition. Alignement fort avec Deming et Hubbard : tu juges le système, pas la métrique en isolation.
Limite
L'utilisateur novice reste sans repère. « Fréquence de déploiement élevée, c'est quoi — par semaine, par jour, par heure ? »
Coût
Nul côté éditorial. Aucun champ ajouté, aucune donnée à saisir.
Réversibilité
Maximale. On peut basculer vers B ou C plus tard sans rien casser.
Rendu fiche o1 (DORA, repères existent)

🌉 Indice de confiance organisationnelle

ICP · Stratégique · Précaution

Score agrégé à trois niveaux (direction, management proximité, pairs), mesuré par enquête annuelle.

Anti-pattern : cible « toujours plus haut », scoring nominatif, inclusion dans la rémunération variable des managers.

// Aucun champ ajouté.
// La fiche n'a pas de `reperes`,
// `executer(filtre)` ignore la notion.

filtre: {
  niveau, tags, cadres,
  fiabiliteMin, maturiteMax, …
}
// rien de plus
B Repères sectoriels indicatifs, typés « à calibrer »

Un champ optionnel reperes sur la fiche, réservé aux métriques dont un référentiel reconnu publie des bandes (DORA, NIST, SLA ITIL). Clairement étiqueté : « Repères industriels — à adapter à ton contexte ». Les fiches sans référentiel légitime n'en portent aucun.

Avantage
Pédagogique. Le novice se projette concrètement. L'étiquette « à calibrer » désamorce la lecture-cible.
Limite
Risque Goodhart résiduel : les bandes peuvent être lues comme des cibles même étiquetées indicatives. Discipline d'affichage nécessaire (collapsé par défaut, mention source obligatoire).
Coût
Modéré. Travail éditorial sur ~15 fiches où un référentiel reconnu existe (DORA×4, ITIL/FCR, NIST, NPS/CSAT standards). Reste zéro sur les autres.
Réversibilité
Bonne. Le champ est optionnel ; on peut le retirer ou le rapatrier vers C plus tard.

🌉 Indice de confiance organisationnelle

ICP · Stratégique · Précaution

Score agrégé à trois niveaux, mesuré par enquête annuelle.

Pas de repère publié. Cette métrique s'observe dans le temps — la variation fait sens, le niveau absolu n'en a pas.
Contrat CM.RequeteMetriques
// Champ optionnel ajouté sur la fiche.
// `executer(filtre)` ne le filtre pas —
// c'est un champ d'affichage.

fiche: {
  …,
  reperes?: {
    source: "DORA 2024",
    bandes: [
      { nom, seuil, qualif }
    ]
  }
}
C Seuils paramétrables côté panier

Le domaine reste nu. Quand une métrique entre dans le panier de l'utilisateur, il peut y attacher ses propres bandes — ses cibles, sa qualification, son contexte. Le catalogue est pur ; la personnalisation vit dans le panier.

Avantage
Aucune imposition depuis le cœur. L'outil devient actionnable sans contaminer le domaine. Forme forte de l'esprit Deming : les cibles appartiennent au contexte.
Limite
L'utilisateur novice n'a pas de repère pour poser ses seuils. Chantier UX lourd côté panier. Élargit le périmètre du chantier 14 qui dit explicitement « panier non touché ».
Coût
Nul côté domaine. Élevé côté UX panier (champ, calibrage, validation, sauvegarde).
Réversibilité
Bonne côté domaine. Moyenne côté panier — une fois les seuils personnels saisis, les retirer est une régression perçue.

ICP · Stratégique · Précaution · Dans ton panier

⚠ Champ paliers désactivé pour cette métrique

La fiche porte un flag seuilsBrides: false. Propose plutôt d'observer la variation annuelle — ouvrir la vue tendance.

Contrat CM.RequeteMetriques
// Côté domaine : un seul flag,
// pour bloquer les métriques-observation.

fiche: {
  …,
  seuilsBrides?: boolean
}

// Côté panier (hors chantier 14) :
entreePanier: {
  idFiche,
  paliers?: [
    { nom, seuil }
  ]
}

Ce qu'il reste à trancher

Question principale. A, B, ou C — ou une combinaison ?

Combinaison possible évoquée. A aujourd'hui + B plus tard, sur les seules fiches à référentiel reconnu, décidé fiche par fiche lors du passage éditorial. C resterait une évolution future du panier, hors chantier 14.

Sous-questions si on retient B.

  • Quelles sont les ~15 fiches à référentiel reconnu ? Liste à figer.
  • Quelle structure de reperes : bandes fixes à 3–5 paliers, ou flexible (array de {nom, seuil, qualif}) ?
  • Collapsé par défaut ou toujours visible ? Quelle place typographique dans la fiche ?

Sous-questions si on retient A.

  • Garde-t-on quand même un champ seuilsBrides pour les métriques-observation type s8 — afin de préparer C plus tard ?
  • Ajoute-t-on une mention pédagogique dans la fiche (« comment poser tes propres paliers »), ou rien du tout ?
