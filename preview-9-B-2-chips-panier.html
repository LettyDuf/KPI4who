Preview 9.B.2 — Chips sur la carte du panier

Même contenu dans les deux panneaux — 7 fiches réelles du référentiel choisies pour couvrir les 7 types (IGO, ICP, ICC, ORC, DORA, FLUX, II) et les 3 fiabilités (fiable, précaution, risque). Deux axes de différence, tout le reste est identique.

Ce que 9.B.2 demande

Enrichir la carte du panier (variante C miroir, livrée en 9.B.1) avec trois chips sous le nom : type, niveau, fiabilité. Lecture via CM.Referentiel.chercher(ficheId) + réutilisation du ViewModel CM.FicheViewModel.from déjà en place.

Ce preview tranche deux décisions : le style du chip type (doux vs plein) et la forme de la fiabilité (pastille vs emoji). Le chip niveau est identique dans les deux panneaux — pas de désaccord à ce niveau.

Les 2 décisions qui diffèrent entre A et B

Le chip niveau, la structure miroir, le × retirer, le nom cliquable, la largeur de carte — tout ça est identique dans les deux panneaux. Ce qui change :

Décision Panneau A — iso-mockup Panneau B — iso-reste-de-l'app
Chip type — style Fond teinté clair, texte sombre. Les chips restent discrets, la carte reste lisible même avec plusieurs chips. Fond plein à la couleur du type (CM.Config.TYPES_COULEURS), texte blanc. Vocabulaire visuel identique au chip de l'en-tête du tiroir de fiche (.fi-chip).
Fiabilité — forme Petite pastille colorée (7px) + le mot. Neutre, discret. Emoji ✅ / ⚠️ / ⚡ + le mot. Mêmes glyphes que CM.Config.FIABILITE.icone, déjà utilisés dans le tiroir de fiche et les listes d'indicateurs.
Question séparée, non bloquante pour 9.B.2. Les deux panneaux montrent le code référentiel à 3 lettres (IGO, ICP, DORA…). Si tu veux à terme un libellé plus parlant (KGI, KPI, Flux, Qualité…), c'est une décision à part — ça suppose d'ajouter un mapping TYPES_LABELS_COURTS à CM.Config et de décider quel libellé par type. On peut trancher plus tard, sans bloquer le câblage de 9.B.2 ni 9.B.3.
